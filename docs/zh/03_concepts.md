# 使用 FluxGraph 思考 (核心机制与原语)

在跳入如何使用更复杂的 RAG 或通过 A2A 协议连接远程大集群之前，你必须先搞懂 FluxGraph 底层极度抽象然而纯粹干净的**状态机哲学**。

不同于以往你通过回调函数链条 (`onReply`, `onToolCalled`) 或面条式的 `if-else` 去编排 LLM 的逻辑，**FluxGraph 的设计核心理念就是：将控制流 (流转) 与数据结构 (记忆) 完全解耦**。

理解以下三大原语机制是成为进阶工程师的基石：

## 1. 原语：只读纯函数与唯一的 `AgentState`

在 FluxGraph 面向领域的开发范式中，整个系统的“血液”只有一个，叫做 [AgentState](../../core/agent_state.go)（代理状态）。

* **单向不可变**：你不能也不应该随时随刻去向全局某个变量 `.append()`。你拿到这一回合最全旧记录的快照参数，做处理后，返回**新叠加（只增不减）**的结果镜像交由引擎保管。
* **包裹万象**：由于多模态模型对上下文有着不同源格式要求（比如你既给了一张图片又挂了一些历史 prompt），`AgentState` 不只是简单的 String 拼接。它内部严密的挂载了 `Message`（消息头）与 `Part`（消息零件，如图片块、请求体块）。

> [!NOTE]
> 这也就是为什么当我们重试失败节点的请求 (Retry / resilience 机制) 或者是触发“中断 (`Interrupts`) 等待审核” 时，FluxGraph 是如此游刃有余。因为 `State` 落库之后，引擎下次拉起的只不过是从最后一次安全状态的镜像开始继续纯函数演算！

## 2. 图的经纬度：`Node` 与 `Edge` 的组合艺术

为了让每一次计算都是确定和可观测的，你的整个业务流水线是一座拓扑图 (`Graph`)。

- **Node (处理节点)**
  Node 是 `State` 的提纯器！比如：
  - 一个查询天气大模型的 LLM Node。
  - 一个运行 Python 代码的沙盒 Node。
  - *甚至一个由其它语言编写的 A2A gRPC 分布式 Node（下一章讲）。*
  它们通过接口约定：它会拉取当前的 `AgentState` 甚至包含历史工具调用结果，输出新的一截计算残影返回。

- **Edge (导航边线)** 
  如果你希望根据当前结果的分析走向分支（比如：假如回答已经触及成功标志 `Finish` 则执行退场，或者比如判断大模型调用返回了 ToolCalls 就走向 `SandboxNode` 加载沙盒执行对应函数并回环），这在传统的黑盒 LLM 里需要疯狂 `try catch` 并 `while` 嵌套重定向。
  
  在 FluxGraph 里，这是优雅无比的一根有向图的边界判定线（`Conditional Edge`）。

## 3. 把掌控权交还引擎 (Engine)

当你画好了所有的 Node 并打通所有的 Edge 后，你将得到一个包含静态流图结构的 `Graph` 对象。

不要去写死循环去轮询状态！

引擎 (`engine.NewEngine(g)`) 包揽了你的一切：
* 注入强大的 Hook 生命流。
* 进行超时阻截与宕机重启补偿。
* 在节点流转的瞬间自动推平所有监控数据到你的 `Prometheus` 指标池面板与 `OTel`/`Jaeger` 分布式追踪系统里。
