# 概述

欢迎来到 FluxGraph 官方中文指南。如果你正在寻找一个能够将 LLM 大模型能力安全、高效地引入到**真实微服务生产环境**中的编排框架，那么你找对地方了。

## 什么是 FluxGraph？

FluxGraph 并非仅仅是对 LLM API 的再一次简单封装。它是一个**生产级、基于图状态机驱动的 Go 原生 Agent 框架**，旨在作为你构建智能体网络和业务管线的“操作系统层”。

通过 FluxGraph，复杂的多 Agent 架构、工具流、多重 RAG 检索管线，全部被解耦为一张有向图（Directed Graph）上的节点流转。每一次逻辑判断、每一轮大模型的思考（ReAct）或是每一次函数工具调用，都作为图中的一个独立节点执行，节点之间只通过纯粹的数据状体 (`State`) 进行交互。这种高度确定性的状态机设计，让以前难以调试、极其容易“幻觉脱轨”的黑盒 Prompt 链条，变成了清晰可测、可控的回调节点。

### 为什么选择 FluxGraph （对比传统 Python 框架）？

当前业界虽有如 LangGraph、AutoGen 等非常优秀的 Python 探索框架，但在面向企业核心业务网关或高频事务系统时，你往往会遇到所谓的“深水区”挑战：

1. **Python 并发瓶颈与资源开销**：在数百个 Agent 实例并发的场景下，传统框架极易遇到协程与线程调度的瓶颈。而 FluxGraph 利用 Go 语言强大的 goroutine 原生并发加持，带来指数级的吞吐量飞跃。
2. **缺乏真正的“常驻”企业记忆**：演示级的项目通常用一个 list 收拢上下文对付过去。而 FluxGraph 拥有出厂即配的 **双层记忆架构 (Dual-Tier Memory)**。高速请求走 Redis，漫游 RAG 寻址走 PostgreSQL + pgvector 原生向量检索，做到无需用户操心的历史会话持久化。
3. **原生微服务隔离与通信协议**：Python 框架往往局限于“同一个运行黑盒脚本内”的多智能体，而 FluxGraph 内置工业级的 Agent-to-Agent (A2A) 网络通信协议（gRPC/REST）。这意味着，你的每一个图都可以编译成为独立的微服务，像蜂群一样通过标准化网络接口跨实例沟通。
4. **缺失生产护栏**：FluxGraph 深知生产不是儿戏，原生集成了包括 **Global OTel 可观测性探针（链路追踪）**、**Security Guard 安全脱敏护栏**，以及 **Resilience 退避（如 OpenAI 宕机自动跳备用 Anthropic 模型）**。

## 核心架构概览

以下是你在使用 FluxGraph 开发时，将会接触到的几大主要模块：

### 1. 引擎与图 (Engine & Graph)
这是框架的运转心脏。你使用 `graph.NewGraph()` 定义图拓扑，接着将你的业务函数包裹为一个个 `Node` (节点) 添加进去，并利用条件边 (`Edge`) 指导走向。最后，挂载到 `Engine` 这个执行器上驱动无缝流转。

### 2. 状态与上下文原语 (State & Context primitives)
摒弃繁杂的传参。FluxGraph 中的一切数据流动都依赖一条只读并以增量覆盖的流：`AgentState`。它承载了对话的 `Message`，甚至支持包含图像、文本等多种来源混合的流媒体对象 `Part`。

### 3. 微服务网络与 A2A (Microservices & Agent-to-Agent)
这是我们的**最高级商业用法**。利用我们的 `a2a` 库，你能原地把本地 Agent 提升为暴露 gRPC/Webhook 的远程网络实例。你甚至能在自己的图中插入“代理节点 (`Delegate Node`)”，在图中无缝把某个分支任务通过 RPC 路由去隔壁物理机器上的其他语言的微服务。

### 4. 工具中心与沙盒 (Tool Registry & Sandbox)
框架提供了一套高度一致的接口约定。你可以通过 `registry` 将公司内部所有的 OpenAPI、数据库查询 SQL 函数、爬虫模块甚至外部 Python 脚本无缝接入，让 LLM 的触手延伸到现实物理世界。
